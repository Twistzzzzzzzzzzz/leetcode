# 621. Task Scheduler

## 题目

给定一个任务数组 `tasks` 和冷却时间 `n`。

每个时间单位可以：

- 执行一个任务；
- 或者保持空闲 `idle`。

两个相同任务之间至少要间隔 `n` 个时间单位，求完成所有任务需要的最短总时间。

例如：

```text
tasks = [A, A, A, B, B, B]
n = 2
```

一种最短安排是：

```text
A B idle A B idle A B
```

总时间为 8。

## 最初想法需要修正的地方

为每类任务记录 cooldown 的方向是对的，但调度优先级不应该是：

```text
cooldown 最小优先，字母大小次要
```

更准确地说，系统需要回答两个不同问题：

```text
冷却状态：这个任务现在能不能执行？
剩余次数：可执行的任务中，应该先执行哪个？
```

因此使用两个数据结构，各自只负责一件事：

1. 最大堆：保存当前可以执行的任务，剩余次数最多者优先。
2. 冷却队列：保存暂时不能执行的任务，以及它们重新可用的时间。

## 为什么优先执行剩余次数最多的任务

例如当前可执行任务为：

```text
A 剩余 5 次
B 剩余 1 次
C 剩余 1 次
```

如果一直推迟 A，最后很可能只剩许多个 A，中间不得不加入大量 `idle`。

所以应该尽早穿插出现次数最多、最难安排的任务：

```text
在当前可执行任务中，优先选择剩余次数最多者
```

这部分体现的是贪心思想，最大堆负责快速找到这个候选。

## 为什么先统计完整频率

堆中的一个元素代表：

```text
一种任务还剩多少次
```

而不是一个单独的任务实例。

例如：

```text
tasks = [A, A, A, B, B, C]
```

完整频率是：

```text
A: 3
B: 2
C: 1
```

最大堆应保存三个任务类型的剩余次数：

```text
3, 2, 1
```

而不是为六个任务实例分别保存六个 1。

可以在遍历 `tasks` 时统计字典，但应该等频率统计完成后再建堆：

```python
frequency = Counter(tasks)
max_heap = [-count for count in frequency.values()]
heapq.heapify(max_heap)
```

如果一边统计一边修改堆，就需要查找并更新堆中旧的频率；`heapq` 不擅长修改中间的指定元素，这里没有必要增加复杂度。

## 为什么堆里不需要任务名

本题只要求最短总时间，不要求返回具体执行顺序。

所有任务遵守相同的：

- 执行时间；
- 冷却时间；
- 优先级规则。

因此任务名称只是标签。我们只需要保留每一种任务的剩余次数：

```python
max_heap = [-3, -3, -1]
```

即使两个 `-3` 无法区分哪个是 A、哪个是 B，也不影响总时间计算。它们仍然是两条独立的任务类型记录。

当某条记录被弹出执行后，它会离开可执行堆并进入冷却队列，所以同一种任务不会在冷却结束前再次被选中。

以下情况才需要额外保存任务名：

- 题目要求返回具体调度顺序；
- 不同任务有不同冷却时间；
- 不同任务有不同执行时间或优先级；
- 调试时希望看到具体执行了哪个字母。

## Python 中如何建立最大堆

Python 的 `heapq` 是最小堆，因此把频率取负：

```python
max_heap = [-count for count in Counter(tasks).values()]
heapq.heapify(max_heap)
```

例如频率为：

```text
3, 2, 1
```

堆中保存：

```text
-3, -2, -1
```

最小值 `-3` 位于堆顶，对应原频率中最大的 3。

## 执行一次后为什么是 `+ 1`

从堆中弹出的剩余次数是负数。

例如某类任务还剩 3 次：

```text
remaining_count = -3
```

执行一次后应该剩 2 次，也就是：

```text
-3 + 1 = -2
```

所以代码是：

```python
remaining_count = heapq.heappop(max_heap) + 1
```

不是 `- 1`。负数频率在任务被执行后，应不断向 0 靠近。

如果结果为 0，说明这一类任务已经全部执行完，不再进入冷却队列。

## 冷却队列保存什么

冷却队列保存：

```text
(ready_time, remaining_count)
```

含义是：

```text
这类任务还剩 remaining_count 次，到了 ready_time 才能重新进入可执行堆
```

不要每秒遍历所有冷却任务并把 cooldown 减一。直接保存绝对恢复时间更清晰，也不需要在遍历 `deque` 时删除元素。

## 为什么恢复时间是 `time + n + 1`

假设在时间 1 执行 A，并且：

```text
n = 2
```

那么：

```text
时间 1：执行 A
时间 2：不能执行 A
时间 3：不能执行 A
时间 4：可以再次执行 A
```

因此：

```python
ready_time = time + n + 1
```

这里的 `+ 1` 来自：当前执行时间也占用了一个位置，后面还要完整间隔 `n` 个时间单位。

## 为什么冷却队列使用 `deque`

每个时间单位最多执行一个任务，所以任务按时间先后进入冷却队列。

所有任务的冷却时间 `n` 相同，因此：

```text
先进入冷却队列的任务，也会先恢复
```

恢复时间自然有序，只需要不断检查队头：

```python
while cooldown and cooldown[0][0] <= time:
    _, remaining_count = cooldown.popleft()
    heapq.heappush(max_heap, remaining_count)
```

使用 `while` 和 `<=` 可以让这个不变量更稳健：所有已经到达恢复时间的任务都会回到堆中。

## 每个时间点的处理顺序

整理后的模拟顺序是：

```text
1. 时间向前移动一个单位。
2. 把已经结束冷却的任务放回最大堆。
3. 如果最大堆非空，执行剩余次数最多的任务。
4. 执行后剩余次数减一。
5. 如果该任务还有剩余，记录它下一次可用的绝对时间。
6. 如果最大堆为空但冷却队列非空，这个时间单位就是 idle。
```

这里不需要专门为 `idle` 写分支。时间已经增加，而本轮没有从堆中执行任务，自然就模拟了一次空闲。

## 执行示例

```text
tasks = [A, A, A, B, B, B]
n = 2
```

初始频率：

```text
A: 3
B: 3
```

为了便于理解，下面写出任务名；实际代码只保存剩余次数。

| 时间 | 本轮操作 | 执行后状态 |
| --- | --- | --- |
| 1 | 执行 A | A 剩 2，时间 4 恢复 |
| 2 | 执行 B | B 剩 2，时间 5 恢复 |
| 3 | `idle` | 没有可执行任务 |
| 4 | A 恢复并执行 A | A 剩 1，时间 7 恢复 |
| 5 | B 恢复并执行 B | B 剩 1，时间 8 恢复 |
| 6 | `idle` | 没有可执行任务 |
| 7 | A 恢复并执行 A | A 完成 |
| 8 | B 恢复并执行 B | B 完成 |

最终总时间为 8。

## 代码

```python
import heapq
from collections import Counter, deque


class Solution(object):
    def leastInterval(self, tasks, n):
        max_heap = [-count for count in Counter(tasks).values()]
        heapq.heapify(max_heap)

        cooldown = deque()
        time = 0

        while max_heap or cooldown:
            time += 1

            while cooldown and cooldown[0][0] <= time:
                _, remaining_count = cooldown.popleft()
                heapq.heappush(max_heap, remaining_count)

            if max_heap:
                remaining_count = heapq.heappop(max_heap) + 1

                if remaining_count < 0:
                    ready_time = time + n + 1
                    cooldown.append((ready_time, remaining_count))

        return time
```

## 必须维持的不变量

### 最大堆

```text
只保存当前已经结束冷却、可以立即执行的任务类型
```

堆顶对应剩余次数最多的可执行任务。

### 冷却队列

```text
只保存暂时不能执行的任务类型
```

每条记录带有重新可用的绝对时间，而且队列按 `ready_time` 递增。

同一种任务不会同时出现在最大堆和冷却队列中。

## 复杂度

设任务总数为 `m`，不同任务类型数为 `u`。题目中任务为大写字母，因此 `u <= 26`。

- 时间复杂度：O(total_time + m log u)
- 空间复杂度：O(u)

`total_time` 包含执行任务和 `idle` 的所有时间单位；每个时间单位只进行常数次队列检查，而每次任务执行最多引起常数次 O(log u) 的堆操作。

因为任务类型最多只有 26 种，堆操作的规模很小。

还可以在堆为空时直接把时间跳到队头的 `ready_time`，避免逐个模拟连续空闲时间；当前版本保留逐秒模拟，更容易理解冷却过程。

## 另一种数学公式解法

这道题也可以只根据最高频率计算答案：

```text
max_frequency = 最高任务频率
max_count = 达到最高频率的任务类型数量

框架长度 = (max_frequency - 1) * (n + 1) + max_count
答案 = max(任务总数, 框架长度)
```

不过当前处于 Heap / Priority Queue 专题，最大堆加冷却队列更适合练习：

- 如何设计堆中状态；
- 如何区分可执行集合和冷却集合；
- 如何通过优先队列实现贪心调度。

## 原代码中的关键问题

### 对负数频率执行了 `- 1`

```python
next_task -= 1
```

会让 `-3` 变成 `-4`，表示剩余次数反而增加。

正确方向是：

```python
next_task += 1
```

### 尝试修改元组

冷却队列中加入的是元组：

```python
(remaining_count, cooldown)
```

元组不可修改，因此 `item[1] -= 1` 会报错。

### 遍历 `deque` 时删除元素

一边遍历队列一边调用 `remove` 容易跳过元素或破坏遍历。保存绝对恢复时间后，只需从队头 `popleft()`。

### 把整个频率字典压入堆

```python
heapq.heappush(heap, -freq)
```

`freq` 是字典，不能取负。应该放回当前任务的剩余次数。

### 额外弹出了一次任务

本轮已经执行过一次 `heappop`，末尾再次无条件弹出会直接丢失另一个任务，并且可能对空堆操作。

## 心得

1. cooldown 负责判断任务是否有资格执行，不负责决定可执行任务之间的优先级。
2. 可执行任务中，优先处理剩余次数最多的类型，避免高频任务被拖到最后产生额外 `idle`。
3. 先统计每种任务的完整频率，再建立调度堆；堆中一个元素代表一种任务类型。
4. 本题只求最短时间，任务字母本身不影响结果，所以堆中无需保存任务名。
5. Python 用负数模拟最大堆，执行一次后应 `+ 1`，让负数向 0 靠近。
6. 冷却队列保存绝对恢复时间，比每秒修改所有 cooldown 更直接。
7. 两个数据结构各自维护清晰的不变量：最大堆存可执行任务，队列存冷却中的任务。
